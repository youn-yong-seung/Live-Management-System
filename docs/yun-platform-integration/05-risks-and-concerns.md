## 🟢 R1. 카카오 OAuth — Resolved (2026-05-11)

**원래 우려**: 한국 사용자 베이스에서 카카오 OAuth 미지원 시 가입 전환율 저하.

**해소된 이유**:
- env.local 분석 결과 yun-platform은 **구글 OAuth만 운영 중** (카카오 OAuth 키 없음)
- 즉 사용자가 이미 "구글 + 이메일 매직링크"로 가입 흐름을 결정한 상태
- Phase 1A에서 카카오 OAuth Custom Provider PoC 불필요

**대신 새로 생긴 함의 (→ R15로 이관)**:
- 한국 사용자에게 구글만 제공하면 전환율 손실 가능
- → 이메일 매직링크가 사실상 메인 가입 경로
- → 카톡 알림톡 안에 매직링크 URL을 박는 흐름이 더 중요해짐 (Phase 1B 산출물에서 강조)

## 🔴 R2. 라이브 신청 → 자동 가입은 개인정보 보호법 이슈

**왜 문제**:
- 한국 정통법: 회원가입은 별도 동의 항목 필수. 라이브 신청 동의 ≠ 회원가입 동의
- 신청자 모르게 회원으로 등록하면 행정처분 + 신뢰 손상
- 마케팅 수신 동의 / 제3자 제공 동의도 분리 항목 필요

**완화**:
- 라이브 신청 폼에 체크박스 명시: "윤자동 클래스 회원가입 / 자료실·커뮤니티 이용 (선택, 미동의 시에도 라이브 참석 가능)"
- 매직링크 메일 본문에 "이 링크 클릭이 곧 회원가입에 해당함" 명시
- 개인정보처리방침 + 이용약관 페이지 우선 작성 (Phase 1A 산출물에 포함)
- 솔라피 알림톡 템플릿 재승인 신청 (메시지에 "회원가입" 언급 추가 시 카카오 심사 필요)

## 🔴 R3. 운영 중인 라이브 시스템에 대규모 변경 — 신청 폼 / cron 깨지면 즉시 사고

**왜 문제**:
- 매주 라이브 진행 중. 신청 폼이 5분만 다운돼도 100-200명 신청자 일부 손실
- Solapi 알림톡 cron은 1분마다 동작 — 인증 미들웨어 추가하다가 cron 인증 실패하면 알림톡 발송 중단 → 신청자에게 안내 누락
- Live는 현재 테스트 코드가 거의 없음 (확인 필요) → 회귀 테스트 자동화 부재

**완화**:
- **Phase 1 작업은 라이브 직후 (1-2일차)에만** — 다음 라이브까지 4-5일 안전 마진 확보
- Phase 1 시작 전 Playwright 회귀 시나리오 최소 3개 작성:
  1. 라이브 신청 → 알림톡 발송 → 신청자 목록 노출
  2. /admin 접근 (현재 인증 없는데 인증 추가 후에도 동작?)
  3. cron 엔드포인트 호출 → 정상 처리
- DB 마이그는 down 스크립트 같이 작성 (롤백 가능)
- 변경은 feature flag 뒤로 (예: `ENABLE_AUTH=true`) → 운영 인증 없이도 기존 흐름 동작

## 🟠 R4. "Phase 1 깔때기"는 가설 — 검증 메커니즘 없이 시작하면 위험

**왜 문제**:
- "라이브 신청자에게 매직링크 보내면 가입할 것" — 가설이지 사실 아님
- 측정 안 하면 Phase 1 끝나고 "왜 가입자 안 늘지?" 사고 발생
- 가능한 이탈 원인: 이메일 안 봄, 매직링크 만료, 카톡만 확인하고 메일 무시, 게이트가 짜증나서 떠남

**완화**:
- Phase 1 산출물에 **메트릭 대시보드** 포함:
  - 라이브 신청 수 / 매직링크 발송 수 / 클릭 수 / 가입 완료 수 (단계별 funnel)
  - 자료실 다운로드 시도 / 비회원 게이트 노출 / 가입 전환 / 다운로드 완료
- Phase 1 끝나고 1주 운영 후 funnel 분석 → 약한 단계 보강 후 Phase 2 진입
- 매직링크는 카톡 알림톡과 함께 발송 (사람들이 카톡을 더 자주 봄)

## 🟠 R5. yun-platform 운영 데이터 처리 미정

**왜 문제**:
- yun-platform이 이미 가입자 / 게시글 / 자료를 가지고 있다면 통합 시 데이터 마이그 + 안내 + 도메인 리다이렉트 필요
- 안내 없이 도메인 바뀌면 기존 사용자 신뢰 손상
- 가입자가 0이면 마이그 작업 자체가 불필요 — 일정 차이 큼

**완화**:
- [04-open-questions.md](./04-open-questions.md) Q4 답 받기 (가입자 / 게시글 수 / 도메인)
- 가입자 ≥ 50명이면 마이그 + 안내 메일 + 도메인 리다이렉트 Phase 5로 별도 일정
- 가입자 0~10명이면 마이그 생략, yun-platform 즉시 종료 검토

## 🟠 R6. 통합 기간 동안 두 시스템 동시 운영 부담

**왜 문제**:
- Phase 1~5 모두 끝나려면 6-10주 예상
- 그동안 yun-platform에서 버그 발견되면 거기도 고쳐야 함 → 작업 2배
- 사용자가 양쪽 어디로 가야 할지 혼란

**완화**:
- yun-platform은 통합 시작 시점에 **신규 기능 추가 동결** (버그픽스만)
- yun-platform 메인 페이지에 "곧 윤자동 클래스로 통합 예정" 배너 추가
- Live 도메인 결정 (yunjadong-class.com 등) → Phase 1 끝나는 시점에 사용자 안내

## 🟠 R7. 기술 스택 호환성 함정

**왜 문제**:
- React 버전: Live는 18, yun-platform은 19 → 일부 컴포넌트 / hooks API 차이
- Tailwind: Live는 v3, yun-platform은 v4 → `@theme` syntax / 토큰 정의 방식 다름 (Phase 1A의 디자인 토큰 추출 시 변환 필요)
- shadcn 버전: Live는 (확인 필요), yun-platform은 4.x + base-ui 혼용
- TipTap을 React 19에서 검증되었지만 React 18 SPA에서 일부 ref 패턴 다를 수 있음
- Vercel AI SDK는 streaming을 Edge Runtime 가정 → Express에서 쓰려면 응답 스트림 핸들러 직접 작성

**완화**:
- Phase 1A에서 React/Tailwind 업그레이드 여부 결정 (한 번에 v4 + React 19로 갈지)
- 단, 기존 라이브 시스템 코드 (Drizzle 스키마 / wouter 라우팅 등) React 19에서 정상 동작 확인 필요
- 컴포넌트 이식 전 패키지 버전 매트릭스 정리 (Phase 1A 산출물)

## 🟠 R8. 인증 미들웨어가 admin 전체에 영향

**왜 문제**:
- 현재 `/admin`은 인증 없이 노출됨 (확인 필요)
- 인증 도입 시 admin 전체 접근 패턴 변화 → 기존 운영자(사용자 본인) 작업 흐름 깨짐
- Solapi cron은 `/api/cron`로 들어오는데 인증 추가하면 cron secret 별도 처리 필요

**완화**:
- admin은 `role='admin'` 사용자만 통과하는 단순 미들웨어
- cron은 `Authorization: Bearer <CRON_SECRET>` 검증으로 별도 트랙
- Vercel Cron Job 헤더 (`Vercel-Cron`)로 1차 필터링 가능

## 🟡 R9. 비용 — Supabase / 메일 / 스토리지

**왜 문제**:
- Supabase Free 50K MAU — 라이브 매주 100-200명 누적 시 4-5개월 후 도달 가능
- 매직링크 메일 발송: 라이브당 100-200통 + 게이트 가입 → 월 800-1500통 → Resend 무료 100/일 한도 초과 가능
- 스토리지: PDF + 커뮤니티 첨부 → Supabase Storage Free 1GB 빠르게 채워짐

**완화**:
- 비용 모니터링 대시보드 (Phase 1 산출물에 포함)
- Free → Pro 전환 시점 (~월 $25) 사용자에게 알림
- Resend 유료 플랜 ($20/mo, 50K 메일) 미리 결제 vs 무료 한도 운영

## 🟡 R10. 매직링크 보안

**왜 문제**:
- 매직링크 URL 유출 시 계정 탈취 (이메일 해킹, 카카오 메시지 캡처 등)

- 한 명이 라이브 신청 폼에 다른 사람 이메일 입력 가능 — 그 사람 명의로 가입됨

**완화**:
- 매직링크 만료: 15분 권장
- 1회 사용 후 폐기
- 가입 완료 후 본인 인증 단계 추가 검토 (전화번호 SMS — 솔라피 활용)

## 🟡 R11. UX — 너무 많은 단계 = 이탈

**왜 문제**:
- 사용자 동선: 라이브 시청 → 자료 받으러 옴 → 다운로드 누름 → 로그인 게이트 → 매직링크 메일 보냄 → 메일 열어 클릭 → 다시 사이트 → 다운로드
- 단계가 많으면 모바일에서 70%+ 이탈

**완화**:
- "이미 라이브 신청한 분"이면 신청 폼에서 받은 이메일로 자동 매칭 → 다운로드 페이지에서 OTP만 입력해도 통과
- 비회원에게도 PDF 미리보기 첫 페이지는 노출 — 가입 가치 보여줌
- Phase 1B에서 신청 → 매직링크가 동시 발송되면 라이브 끝나고 자료 받으러 올 때는 이미 회원

## 🟡 R12. 결정 lock-in (Supabase 종속)

**왜 문제**:
- Supabase로 깊이 들어가면 Auth / RLS / Storage / Functions 다 종속 → 빠질 때 비용 폭발
- 사업이 커져서 자체 인프라로 가야 할 시점이 오면 마이그 큰 작업

**완화**:
- 비즈니스 로직은 Supabase RLS에 의존하지 말고 Express 핸들러에서 권한 체크 (이중)
- Drizzle ORM은 Postgres라 어디든 옮길 수 있음 → Supabase Postgres만 쓰고 Auth/Storage 분리 옵션 열어둠
- 6개월마다 종속 점검

## 🟡 R13. 사용자(본인) 의사결정 병목

**왜 문제**:
- Phase별로 결정 필요 포인트 많음 (Q1-Q5 외에도 디자인 / 카피 / 도메인 / 가격 등)
- 사용자가 다른 일(라이브 진행, 영상 촬영, 카톡방 운영) 바빠서 답이 늦어지면 작업 정체

- 매주 1회 결정 회의 시간 짧게라도 잡기 권장

## 🟠 R15. Vercel 프로젝트 교체 — 환경/OAuth/빌드 설정 전환 시 다운타임 위험

**왜 문제** (ADR-007 결정에서 파생):
- 기존 yun-platform Vercel 프로젝트(`yun-landing-dev`)에 Live 코드를 push하면 빌드 명령어, 환경변수, OAuth Redirect URI, vercel.json 라우팅이 한꺼번에 바뀜
- 한 번에 다 바꾸면 yun-platform 기존 사용자가 사이트 접속 시 빈 페이지 / 500 에러 / OAuth 실패 등 사고
- 특히 OAuth Redirect URI는 Google Cloud Console에서 사전 등록 필요 — 등록 누락 시 로그인 전체 마비

**완화**:
- **Vercel Preview Deployment를 풀 활용** — Live 코드를 별도 git 브랜치로 푸시 후 preview URL에서 검증, OK일 때만 production promote
- Google Cloud Console에 Live의 새 콜백 URL 미리 등록:
  - `https://yun-landing-dev.vercel.app/api/auth/google/callback` (이미 있을 수도)
  - `http://localhost:5173/api/auth/google/callback` (Live dev 포트)
  - Preview URL 패턴: `https://*-ceo-yunjadongcos-projects.vercel.app/api/auth/google/callback` (와일드카드 등록 가능 여부 확인)
- Live의 `vercel.json` 빌드 명령은 monorepo 구조(pnpm + esbuild) — yun-platform의 next 빌드와 완전 다름. 빌드 명령어 전환은 production promote 단계에서 한 번에
- 전환 직전 yun-platform 코드의 git tag 찍어두기 (`yun-platform-final`) — 롤백 가능하게

## 🟠 R16. 한국 사용자 베이스 + 구글 OAuth 단독 = 가입 전환율 저하 우려

**왜 문제** (R1 해소 후 새로 등장):
- 한국에서 구글 계정만 보유한 사용자는 소수 (대부분 네이버, 카카오 메인)
- 라이브 카톡방에서 유입된 사용자는 카카오 가입을 기대 → 구글 가입 버튼만 보이면 이탈
- 매직링크 이메일이 카톡으로 와도 사람들이 잘 안 열어봄

**완화**:
- Phase 1B의 솔라피 알림톡 본문에 매직링크 URL을 직접 노출 (긴 URL이라도)
- 라이브 신청 폼 자체를 "원클릭 가입 폼"으로 — 이메일 입력 직후 자동 가입 + 매직링크는 백업 인증 수단
- Phase 2 이후에라도 카카오 OAuth Custom Provider 구현 검토 — yun-platform `supabase/functions/embed` 코드 분석해서 카카오 토큰 교환 라우트 가능 여부 확인
- (옵션) 네이버 OAuth Custom Provider도 같이 검토

## 종합 — 지금 가장 위험한 3가지 (2026-05-11 갱신)
